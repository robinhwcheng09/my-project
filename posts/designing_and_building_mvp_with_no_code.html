Designing and building MVP with no-code tools

Back in November 2021, we wanted to launch an MVP for Arc, where remote companies meet great developers. The idea was to allow companies to browse through developers and invite them to interview right away.

The first step is to plan out the process and decide on the tools for implementation.

  1. Client views list of developer
  2. Client invites the developer to interview
  3. Developer sees the interview request
  4. Developer responds to the interview request
    1. If accepted, introduce and prep both parties for next step
    2. If rejected, share update with client

With the process listed out, we can think about the implementation. What we found helpful is to break the process into smaller steps and list the important features of each step. This is great for project management and lets us focus on the problem we’re solving for each step.

  1. A list for client to browse
  2. Inviting developer to interview
  3. Updating developer about the interview request
  4. Managing Interview Requests: Developer Responses and Results
  5. Recording and sharing the result

A list for client to browse

To begin, the client will see a list of available developers and have the ability to invite them to an interview. In our first iteration, we used Notion, but found it difficult to maintain and automate.

For our final version, we utilized Spreadsimple, a tool that allows you to build websites using Google Sheets. By populating the Google Sheet with developer information, Spreadsimple generates a user-friendly UI with each developer having their own card and information. This method was perfect for our use case because it allowed us to easily populate the Google Sheet with developer information.

Using Zapier, we collected important information such as the developer's Arc profile URL, short summary, location, and tech stacks, which were then inserted into the Google Sheet. From there, Spreadsimple generated a UI for each developer with their own individual card and information

Inviting developer to interview

On the developer list, each candidate has a brief introduction, their key skills, and a link to their Arc profile, which provides a more detailed overview of their experience. To streamline the interview process, we added an "Invite to Interview" button on each Arc profile page. Clicking this button opens a Typeform that prompts the client to provide their company and role information, which is shared with the developer via Slack. We use Zapier to record and organize these interests on a separate Google Sheet, enabling us to track developer responses and conduct analysis.

The final result is a streamlined process that allows clients to easily invite developers for interviews and provides developers with clear information about the role and company. The use of Typeform and Slack integration enables the process to be automated and track responses efficiently.

Updating developer about the interview request

On the developer end, we use our developer community on Slack. At the beginning of each batch, we create private channels for each developer, which only they have access to. Whenever a client submits an interview request, a Slack message is sent to the developer's private channel with the client’s company and role info. This allows the developer to review the request and decide whether to accept or turn it down.

Implementing the feature for developers to receive interview requests in their private channel was challenging. Initially, we used Zapier's emoji-added trigger, which only listens to one channel at a time, meaning a new zap had to be set up for each private channel. However, this approach was not scalable. After much research, we found a solution that helped us automate all of this.

We use Zapier to record new developer signups on Google Sheets and create private channels for them on Slack. To listen to all emojis added, we integrate Slack's API with an additional tool called Integromate (now Make) because Zapier's Webhook cannot communicate with Slack API in the way we need it to.

Managing Interview Requests: Developer Responses and Results

With Slack's API and the private channel ID stored on Google Sheets, we are able to identify and record the developer's decision regarding the client's interview request. When a developer reacts with an emoji to the interview request message, the corresponding emoji is recorded on Google Sheets, allowing us to keep track of each developer's decision.

To manage interview requests, we use Zapier to record a developer's decision on Google Sheets and send an email to both parties with details for the next step when a developer accepts the interview request. When a developer declines the invitation, Zapier records their decision on Google Sheets and sends an email to the client to update them on the status.

Learnings and conclusion

We encountered a problem when our Zapier workflow was unable to identify the Slack channels recorded on Google Sheets, resulting in clients not receiving any responses. The issue was related to how Google Sheets was changing the last digit to 0 when storing the timestamp in epoch format. To resolve this, we started storing the timestamp in string format, which proved to be a quick solution.

This MVP was accomplished in under a week, demonstrating the power of no-code tools. However, the biggest challenge we faced was understanding the intricacies of the various tools we were integrating.
